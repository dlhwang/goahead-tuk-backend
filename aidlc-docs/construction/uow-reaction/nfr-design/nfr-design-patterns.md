# UOW-REACTION NFR Design Patterns

<!-- markdownlint-disable MD060 -->

## 범위

`UOW-REACTION`은 반응 선택과 해제, 활성 선택 저장, 단일 고해 집계
port의 트랜잭션 및 무결성 패턴을 정의한다.

## 확정된 패턴

| 관심사 | 설계 패턴 | 구현 및 검증 책임 |
| ------ | --------- | ----------------- |
| 동시 선택 | 관계형 데이터베이스 유일 제약을 최종 보장으로 사용하는 멱등 삽입 | `(confession_id, device_key, reaction_type)` 유일 제약을 두고 중복 삽입 충돌은 선택 완료로 변환한다. |
| 선택/해제 | 짧은 트랜잭션의 목표 상태 명령 | `PUT`은 활성 상태를 보장하고 `DELETE`는 행 부재 상태를 보장한다. |
| 집계 | 활성 선택 행의 파생 집계 | `Confession`에 타입별 count를 저장하지 않으며 집계 port가 count를 계산한다. |
| 자기 반응 거부 | 저장 전 작성자 관계 검증 | 원문 기기 식별을 노출하지 않고 `403` 경계로 전달한다. |
| 확장성 | H2 demo 단일 인스턴스 MVP와 영속 DB 전환 경계 | demo는 데이터 초기화를 허용하며, 영속 production 전환 후에도 동일 저장소 제약을 유지한다. |

## 성능 및 신뢰성 결정

- 반응 변이와 단일 집계의 `p95 <= 500 ms`, 변이
  `20 requests/second`는 설계 목표다.
- 사용자 결정에 따라 정량 목표는 MVP 승인 차단 기준이 아니다.
  저장소/통합 테스트와 제한된 동시성 테스트로 경쟁 안전성과
  집계 무결성을 먼저 검증한다.
- 데이터베이스 실패는 성공 응답으로 변환하지 않는다. 유일 제약
  충돌 중 동일 선택의 이미 존재함이 확인되는 경우만 멱등 성공으로
  다룬다.

## PBT 및 검증 패턴

- 임의의 선택/해제 명령 시퀀스를 단순 set 모델과 비교한다.
- 같은 `(confessionId, deviceKey, reactionType)`의 활성 행 수가
  언제나 `0` 또는 `1`임을 검증한다.
- 집계 count가 활성 선택 행의 수와 항상 일치함을 검증한다.
- 경쟁 요청은 예제 기반 제한 동시성 통합 테스트로 유일 제약과
  멱등 오류 변환을 함께 검증한다.
- Kotest 실행 실패 시 seed와 축소 입력을 기록한다.

## 보안 연계

- raw `X-Device-Id` 보호와 변이 API throttling은 `UOW-SECURITY`가
  제공하는 공통 컴포넌트를 사용한다.
- 식별 입력 형식, 허용 타입 allowlist 및 자기 반응 거부는
  Reaction 명령 경계에서도 테스트한다.
