# 고해 반응 MVP 요구사항

## 의도 분석

- **사용자 요청**: 익명 사용자가 댓글이나 자유 텍스트 대신 허용된
  긍정적 반응을 고해에 남길 수 있는 백엔드 기능을 제공한다.
- **요청 유형**: 사용자 대상 백엔드 기능 추가 및 기존 초기 구현 교체.
- **변경 범위**: `confession` API, application, domain, infrastructure,
  데이터 저장 형태, 조회 집계, 검증과 테스트.
- **복잡도**: 중간 이상. 반응 선택은 익명 기기별 멱등 상태이고,
  보안 및 property-based testing 확장을 이번 작업에 적용한다.
- **요구사항 깊이**: 표준. 새 API 계약과 데이터 무결성, 남용 방지 및
  테스트 정책을 명시한다.

## 사용자 가치

고해를 읽은 사람은 공개 댓글을 남기지 않고도 정해진 지지 표현을
선택할 수 있다. 서비스는 자유 텍스트나 부정적 표현을 허용하지 않고,
고해 작성자에게 익명 참여자들의 지지를 타입별 집계로 보여줄 수 있다.

## 기능 요구사항

### FR-1 허용된 반응 타입

서비스가 허용하는 MVP 반응 타입은 다음 세 가지뿐이다.

| 타입       | 표시 의미  |
| ---------- | ---------- |
| `PRAY`     | 기도할게요 |
| `COMFORT`  | 토닥토닥   |
| `TOGETHER` | 함께할게요 |

API는 이 목록 외 타입을 반응 입력으로 수락하지 않는다. 댓글이나 자유
텍스트 반응 입력은 제공하지 않는다.

### FR-2 익명 반응 선택 주체

반응 변경 API는 `X-Device-Id` 헤더를 필수 입력으로 받는다. 이 값은
익명 기기의 반응 선택 상태를 구별하는 데 사용하며 API 응답으로 노출하지
않는다.

### FR-3 선택 집합 모델

한 기기는 한 고해에 각 반응 타입을 독립적으로 선택하거나 해제할 수
있다. 예를 들어 동일한 기기가 동일 고해에 `PRAY`와 `COMFORT`를 동시에
선택할 수 있다.

동일한 `confessionId`, `deviceId`, `reactionType` 조합에는 최대 한 개의
활성 선택만 존재한다.

### FR-4 반응 선택 API

다음 API는 지정 타입의 반응을 선택된 상태로 만든다.

```http
PUT /api/confessions/{confessionId}/reactions/{type}
X-Device-Id: {deviceId}
```

- 고해가 존재하고 타입 및 기기 식별자가 유효하면 `204 No Content`를
  반환한다.
- 이미 선택된 타입에 대한 반복 `PUT`도 `204 No Content`를 반환하며
  중복 선택을 생성하지 않는다.

### FR-5 반응 해제 API

다음 API는 지정 타입의 반응을 해제된 상태로 만든다.

```http
DELETE /api/confessions/{confessionId}/reactions/{type}
X-Device-Id: {deviceId}
```

- 고해가 존재하고 타입 및 기기 식별자가 유효하면 `204 No Content`를
  반환한다.
- 이미 해제된 타입에 대한 반복 `DELETE`도 `204 No Content`를
  반환한다.

### FR-6 반응 집계 조회

고해 단건 및 목록 조회 응답은 각 허용 반응 타입의 활성 선택 수를
집계해 제공한다. 아직 선택되지 않은 허용 타입도 `0`으로 표현할 수
있어야 한다.

### FR-7 존재하지 않는 고해

반응 선택 또는 해제 대상 고해가 존재하지 않으면 기존 고해 조회 오류
정책과 일관된 `404 Not Found` 응답을 반환한다.

### FR-8 입력 오류

지원하지 않는 반응 타입, 누락되었거나 유효하지 않은 `X-Device-Id`,
유효하지 않은 고해 식별자는 안전한 `400 Bad Request` 응답으로
거부한다.

## 데이터 및 도메인 요구사항

### DR-1 별도 반응 도메인 및 테이블

반응 선택은 `Confession` 테이블에 `prayer_count`,
`comfort_count`, `together_count`와 같은 컬럼을 추가해 저장하지
않는다. 반응 타입 확장에 대응할 수 있도록 별도 반응 선택
도메인/테이블로 관리한다.

### DR-2 기기별 선택 저장

타입별 집계 행만 저장하는 기존 초기 구현은 동일 기기의 중복 선택을
막거나 해제를 정확히 처리할 수 없다. 저장 데이터는 최소한 다음 값을
구분할 수 있어야 한다.

- 반응 선택 식별자
- 대상 고해 식별자
- 익명 기기 식별자
- 반응 타입

데이터베이스 제약 또는 동등한 원자적 처리로
`(confession_id, device_id, reaction_type)`의 유일성을 보장한다.

### DR-3 집계 파생

조회에 노출하는 반응 개수는 활성 선택 데이터로부터 계산한다. 타입별
카운트를 `Confession` 엔터티에 직접 저장하지 않는다.

## 보안 및 비기능 요구사항

Security Baseline 확장은 이번 실행에서 활성화되었다. 이 기능 범위에서
다음 요구사항은 구현 및 검증 시 차단 조건이다.

### NFR-1 입력 검증과 안전한 오류

- `X-Device-Id`, `confessionId`, `type`에 allowlist 또는 명시적인
  길이/형식 제한을 적용한다. (`SECURITY-05`)
- 데이터 접근은 JPA의 매개변수화된 쿼리 방식만 사용한다.
  (`SECURITY-05`)
- 오류 응답은 내부 구현이나 저장소 정보를 드러내지 않는다.
  (`SECURITY-09`, `SECURITY-15`)

### NFR-2 익명 식별자 보호

- `X-Device-Id`는 익명 참여자 식별 데이터이며 로그 메시지에 원문을
  기록하지 않는다. (`SECURITY-03`)
- 반응 응답 및 고해 조회 집계는 개별 기기 식별자를 공개하지 않는다.
  (`SECURITY-08`, `SECURITY-13`)

### NFR-3 공개 변이 API 남용 방지

반응 선택 및 해제는 인증 계정 없이 사용하는 공개 변이 API이므로,
기기 식별자만으로 무제한 쓰기가 가능하지 않도록 rate limiting 또는
동등한 throttling 제어를 적용하고 검증한다. (`SECURITY-11`)

### NFR-4 운영 및 프로젝트 수준 보안 확인

암호화 전송/저장, 중앙 로그 및 보존/경보, 공급망 검사와 SBOM,
배포 보안 설정처럼 애플리케이션 기능만으로 완결되지 않는 Security
Baseline 항목은 NFR 단계에서 현재 배포 범위와 증빙을 점검한다.
이번 기능 구현 전에 이미 존재하는 미충족 항목이 확인되면 차단
finding으로 기록하고 처리 범위를 확정한다.

## Property-Based Testing 요구사항

Property-Based Testing 확장은 이번 실행에서 활성화되었다.

### PBT-1 확인할 속성

- 동일한 유효 반응 선택에 대한 반복 `PUT`은 관찰 가능한 선택 집합과
  집계를 바꾸지 않는다. (멱등성)
- 동일한 유효 반응 해제에 대한 반복 `DELETE`는 관찰 가능한 선택
  집합과 집계를 바꾸지 않는다. (멱등성)
- 임의의 유효한 기기별 선택/해제 순서 뒤에도
  `(confessionId, deviceId, type)` 중복 선택은 존재하지 않으며 집계는
  활성 선택 수와 같다. (상태 불변식/모델 기반 검증)
- 허용 타입의 직렬화 및 역직렬화가 추가되는 API 입력 매핑에는 유효
  enum 타입의 왕복 속성을 검증한다. (왕복성)

### PBT-2 프레임워크 및 실행

Kotlin/JUnit 테스트 환경에 맞는 property-based testing 프레임워크를
NFR 단계에서 선정하고, 생성기 품질, shrinking, seed 재현성과 CI
실행 방식을 기록한다. 예제 기반 테스트는 핵심 API 시나리오를 별도로
보호하며 property-based tests로 대체하지 않는다.

## 기존 코드 영향

- 기존 `POST /api/confessions/{confessionId}/reactions`의 카운터 증가
  동작은 확정된 선택 집합 API와 일치하지 않으므로 교체 대상이다.
- 기존 반응 타입 값은 `PRAY`, `COMFORT`, `TOGETHER`로 정리한다.
- 고해 읽기 응답의 타입별 집계 구조는 새 선택 저장소의 집계 결과를
  사용하도록 맞춘다.
- JPA 엔터티와 Spring Data 저장소는 계속 infrastructure 패키지에
  유지한다.

## 범위 제외

- 댓글 작성 또는 자유 텍스트 반응.
- 사용자의 실명, 계정 또는 공개 프로필 노출.
- `Confession` 테이블의 타입별 집계 컬럼.
- 프론트엔드 버튼 구현.

## 수용 기준 요약

1. 유효한 익명 기기는 허용 타입을 `PUT`으로 선택하고 `DELETE`로
   해제할 수 있으며 두 요청 모두 멱등하고 성공 시 `204`를 반환한다.
2. 동일한 기기의 동일 고해/타입 선택은 중복 생성되지 않는다.
3. 한 기기는 하나의 고해에 여러 허용 타입을 동시에 선택할 수 있다.
4. 고해 조회는 개별 기기 정보를 노출하지 않고 활성 선택의 타입별
   집계를 반환한다.
5. 반응 선택은 별도 데이터 모델로 저장되고 `Confession`의 타입별
   카운트 컬럼은 추가되지 않는다.
6. 활성화된 Security Baseline 및 Property-Based Testing 차단 조건을
   설계, 구현, 테스트와 완료 보고에 반영한다.

## Railway H2 Demo 범위 보정

Infrastructure Design 단계에서 사용자는 현재 Railway 배포가 H2 기반
preview이며 Reaction 기능을 빠르게 확인할 일회성 demo 배포를
우선한다고 결정했다.

- demo는 영속 production 배포가 아니며 재시작 또는 재배포 시
  Confession 및 Reaction 데이터가 초기화될 수 있다.
- demo용 H2 profile은 H2 console과 SQL 출력, API docs 노출을
  비활성화해야 한다.
- 입력 검증, 익명 식별 원문 비로그, 안전한 오류, throttling,
  보안 헤더와 PBT는 demo에서도 구현 및 검증 대상이다.
- 저장 암호화, 장기 로그 보존, 보안 경보 및 dashboard 같은 운영
  증빙은 영속 production 배포로 전환하기 전에 완료해야 하며,
  demo 배포는 해당 운영 요구를 충족한 것으로 간주하지 않는다.
